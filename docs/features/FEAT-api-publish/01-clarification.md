# API 发布管理 - 需求澄清文档

**FEAT 编号**：FEAT-api-publish  
**一句话需求**：在数据中台上提供「基于数据连接与表/视图发布 API」的完整能力，支持基于表自动生成与基于表+手写 SQL 两种发布方式，并提供分类、搜索、在线调试与对接文档，供客户正确调用。

**业务背景**：作为中台服务商交付标准化数据中台系统，需同时支持「数据 API」与「业务 API」的发布与治理；本期优先打通基于数据源与表的 API 发布闭环。

**现状/痛点**：缺少统一的数据 API 发布入口；客户不清楚如何基于表暴露接口、如何传参与鉴权；无统一目录、文档与在线调试。

**约束**：底层网关/运行环境依赖现有或将接入的网关组件，本期不自研网关；安全上仅允许 SELECT、表/视图白名单。

**相关系统/模块**：数据源管理（选连接/表）、统一网关或 API 运行时、鉴权/调用方管理（若已有）。

**目标用户/角色**：数据平台管理员、数据/后端开发（发布与配置 API）；业务/前端开发或外部客户（查阅文档、获取密钥、调用 API）。

## 1) 澄清问题清单

### 1.1 【单选】数据对象选择范围（表 vs 视图）

- 问题：基于数据连接选择对象时，本期支持哪些类型？
  - A) 仅表（Table），无 schema 时直接选表，有 schema 时先选 schema 再选表
  - B) 表 + 视图（View），视图与表同等对待（可选表或视图后再发布）
  - C) 表 + 视图 + 存储过程/函数（仅读结果集，不写库）
  - D) 其它：自定义
- **已选**：B）表 + 视图
- 说明：与方案 B 一致，覆盖大部分报表/分析场景，边界清晰。

### 1.2 【多选】API 发布方式

- 问题：本期支持哪几种「发布为 API」的方式？
  - A) 仅「基于表」：从表结构自动带出入参（条件字段）、返回字段与类型，生成标准查询 API
  - B) 仅「基于表 + 手写 SQL」：选表后手写 SQL（参数化），校验仅访问该表或同库白名单
  - C) 同时支持 A 和 B，由创建时选择
  - D) 其它：自定义
- **已选**：C）同时支持「基于表」与「基于表+手写 SQL」
- 说明：与用户需求第 2、3 条一致。

### 1.3 【单选】手写 SQL 的访问范围

- 问题：手写 SQL 允许访问的对象范围？
  - A) 仅允许访问「当前选中的单表」
  - B) 允许访问「当前数据源下已选表 + 同库白名单表/视图」（白名单由管理员维护）
  - C) 允许访问当前数据源下所有表/视图（不推荐，安全风险大）
  - D) 其它：自定义
- **已选**：B）已选表 + 同库白名单
- 说明：平衡灵活性与安全，避免任意库任意表暴露。

### 1.4 【多选】API 分类与发现

- 问题：API 列表/目录支持哪些分类与筛选维度？
  - A) 按业务域/主题域
  - B) 按数据源/连接
  - C) 按标签（可多选）
  - D) 按环境（prod / 非 prod）
  - E) 按状态（草稿/已发布/已下线）
  - F) 关键字搜索（名称、描述、路径）
  - G) 其它：自定义
- **已选**：A + B + C + D + E + F
- 说明：覆盖管理侧与使用侧查找需求。

### 1.5 【单选】在线调试能力

- 问题：本期「在线调试」覆盖到什么程度？
  - A) 在控制台填写入参、选择环境，发起请求并展示状态码与响应 Body（JSON）
  - B) A + 请求/响应 Header、耗时、历史记录（最近 N 次）
  - C) A + 支持保存为示例、生成 cURL/代码片段
  - D) 其它：自定义
- **已选**：A）入参 + 环境 + 请求 + 状态码与响应 Body
- 说明：MVP 先保证「能调通、能看结果」，B/C 可后续迭代。

### 1.6 【多选】对接文档形态

- 问题：让客户知道如何调用，本期提供哪些文档能力？
  - A) 每个 API 独立文档页：请求 URL、Method、入参说明、出参结构、示例、鉴权方式
  - B) 支持导出 OpenAPI 3.0（或等价）文件
  - C) 文档门户：按分类/标签聚合的 API 列表与跳转
  - D) 其它：自定义
- **已选**：A + B + C
- 说明：满足「有对接文档」且便于集成与检索。

### 1.7 【单选】环境与发布状态模型

- 问题：API 的环境与生命周期状态？
  - A) 仅「草稿 / 已发布 / 已下线」三态，不区分测试/生产环境
  - B) 环境区分「测试 / 生产」（或 prod / 非 prod），状态为「草稿 / 已发布 / 已下线」
  - C) 多环境（如 dev/test/pre/prod），每环境独立发布状态
  - D) 其它：自定义
- **已选**：B）环境区分测试/生产 + 三态
- 说明：与数据源管理「prod/非 prod」一致，便于联调与上线。

### 1.8 【单选】版本管理

- 问题：同一 API 是否支持多版本并存（如 v1、v2）？
  - A) 不支持，仅「当前配置即当前版本」，改了就覆盖
  - B) 支持多版本，可标记「当前推荐版本」，旧版本保留一段时间后可下线
  - C) 支持多版本 + 按调用方/Header 灰度路由到不同版本
  - D) 其它：自定义
- **已选**：B）多版本 + 推荐版本 + 可下线
- 说明：方案 B 的增强，避免变更直接破坏已有调用方。

### 1.9 【多选】鉴权与调用方管理

- 问题：本期 API 调用鉴权与调用方管理做到什么程度？
  - A) 不鉴权，仅内网或网关层统一控制
  - B) 为调用方分配 AppKey/Token，API 调用时校验密钥（可选 IP 白名单）
  - C) B + 支持按「调用方」启用/禁用对某 API 的访问
  - D) 接入统一 SSO/OAuth2，不做本模块自有密钥体系
  - E) 其它：自定义
- **已选**：B + C
- 说明：MVP 可落地的鉴权与授权控制，便于文档中写清「如何获取密钥、如何传参」。

### 1.10 【多选】可登录「API 发布管理」的角色

- 问题：本期哪些角色需要直接登录本模块并操作？
  - A) 数据平台管理员 / 系统 OWNER
  - B) 数据工程师 / 后端开发（发布、编辑、下线 API）
  - C) 业务/前端开发或外部客户（仅查看文档、调试、获取密钥）
  - D) 仅管理员，其他角色通过工单或只读文档门户
  - E) 其它：自定义
- **已选**：A + B + C
- 说明：管理员与开发可操作发布；调用方需要文档与调试入口（可为只读/受限）。

### 1.11 【单选】模块内操作权限粒度

- 问题：本期是否做模块内细粒度权限（谁能发布/谁能编辑/谁能只看）？
  - A) 不做细分，能进模块则具备发布/编辑/下线等全部能力（仅区分「管理端」与「文档/调试端」）
  - B) 做基础细分：如「发布者」可增删改 API，「查看者」仅可看文档与调试
  - C) 接入统一权限中心，由上游系统控制操作粒度
  - D) 其它：自定义
- **已选**：B）发布者 vs 查看者
- 说明：与「数据源管理」MVP 略不同，因 API 有明确「使用方」，需要只读/调试权限。

### 1.12 【多选】审计范围

- 问题：本期需要记录哪些操作轨迹？
  - A) API 新增 / 编辑 / 删除（下线）
  - B) 发布 / 下线 / 状态变更
  - C) SQL 或字段映射变更
  - D) 在线调试的发起（操作人、时间、API、结果摘要）
  - E) 敏感数据相关（若返回字段含脱敏配置的变更）
  - F) 其它：自定义
- **已选**：A + B + C + D
- 说明：满足可追溯与合规排查。

### 1.13 【单选】限流与防护

- 问题：本期是否配置 API 或调用方的限流/配额？
  - A) 不做，由网关或基础设施统一限流
  - B) 支持按 API 配置 QPS/日调用上限，超出返回 429 或友好提示
  - C) B + 支持按调用方配置配额
  - D) 其它：自定义
- **已选**：B）按 API 配置上限
- 说明：防止单 API 拖垮库，实现简单；按调用方配额可后续迭代。

### 1.14 【单选】基于表发布时的查询安全

- 问题：「基于表」自动生成查询时，如何避免全表扫描/大量数据拉取？
  - A) 强制至少一个条件字段作为入参，不允许无条件 SELECT 全表
  - B) 允许无条件，但强制分页（page/size），且单页条数有上限（如 100）
  - C) A + B：优先条件约束，若无条件则必须分页且限制单页上限
  - D) 其它：自定义
- **已选**：C）条件或分页二选一，且分页有上限
- 说明：兼顾灵活性与库安全。

### 1.15 【单选】前端形态

- 问题：本期前端形态？
  - A) 仅在数据中台「管理后台」内提供 API 管理菜单（列表、发布、编辑、文档、调试）
  - B) 管理后台 + 独立的「API 文档门户」入口（只读，面向调用方）
  - C) 仅后端接口，由现有系统嵌入/集成
  - D) 其它：自定义
- **已选**：B）管理后台 + 文档门户
- 说明：与 1.10 中「调用方需要文档与调试」一致，文档门户可只读或带简单调试。

## 2) 功能规格草案

### 2.1 目标（in-scope）

- **数据连接与对象选择**
  - 基于已有数据源（数据源管理模块）选择连接；有 schema 则先选 schema，再选表或视图；无 schema 则直接选表或视图。
  - 支持表、视图两种对象类型；列表展示表/视图名、注释等基础信息。
- **基于表发布 API**
  - 选择「查询条件字段」作为入参（名称、类型从表结构带出），选择「返回字段」（全表或勾选列），返回类型从表结构推导。
  - 自动生成查询逻辑；条件支持等值、IN、LIKE、范围等简单算子；安全策略：必须带条件或分页，分页有单页上限。
- **基于表 + 手写 SQL 发布 API**
  - 选表后编写 SQL，支持参数化（如 `:id`）；仅允许 SELECT；仅允许访问已选表或同库白名单表/视图；提交前可做语法与白名单校验。
- **API 管理**
  - 统一 API 台账：名称、路径、方法、环境、状态（草稿/已发布/已下线）、分类、标签、数据源/表来源、负责人等。
  - 支持多版本（如 v1/v2），可标记推荐版本；下线前提示调用影响（若有调用量信息）。
- **分类、搜索与在线调试**
  - 按业务域、数据源、标签、环境、状态筛选；关键字搜索名称、描述、路径。
  - 在线调试：选择环境、填写入参、发起请求，展示状态码与响应 Body（JSON）。
- **对接文档**
  - 每个 API 独立文档页：请求 URL、Method、入参说明、出参结构、示例、鉴权方式（Header/参数）。
  - 支持导出 OpenAPI 3.0；可选文档门户（按分类/标签聚合列表）。
- **鉴权与调用方**
  - 为调用方分配 AppKey/Token；支持 IP 白名单；支持按调用方启用/禁用对某 API 的访问。
- **审计与可观测**
  - 记录 API 新增/编辑/下线、发布状态变更、SQL 或字段映射变更、在线调试发起（操作人、时间、结果摘要）。

### 2.2 非目标（out-of-scope）

- 不自研 API 网关，依赖现有或将接入的网关/运行时。
- 不支持存储过程/函数、复杂多租户计费、OAuth2 完整授权流程、按调用方细粒度配额（本期仅按 API 限流）。
- 不支持 gRPC、GraphQL、WebSocket；仅 HTTP/REST（或 HTTP+JSON）。
- 不做可视化 JOIN 拖拽、复杂 SQL 编辑器（语法高亮/补全可后续迭代）；不做 Mock 服务、Postman 集合导出（可后续迭代）。

### 2.3 用户角色与权限（草案）

| 角色                 | 查看列表/详情 | 发布/编辑/下线 API | 配置 SQL/字段映射 | 在线调试 | 查看文档门户 | 管理调用方/密钥 |
|----------------------|----------------|---------------------|--------------------|----------|--------------|------------------|
| 数据平台管理员       | ✓              | ✓                   | ✓                  | ✓        | ✓            | ✓                |
| 数据/后端开发（发布者） | ✓              | ✓                   | ✓                  | ✓        | ✓            | —                |
| 业务/前端/外部（查看者） | —              | —                   | —                  | ✓（仅己方） | ✓            | —（仅申请/查看己方密钥） |

- 说明：管理端与文档门户可分离；「查看者」仅能使用文档与调试，不能改 API 配置；调用方密钥可由管理员或发布者分配，具体可在 PRD 中再细化。

### 2.4 主流程

1. **选择数据源并选表/视图**：进入「发布 API」→ 选择数据源 → 有 schema 选 schema → 选择表或视图。
2. **基于表发布**：选择条件字段与返回字段 → 配置入参名与默认算子 → 设置 API 名称、路径、分类、环境 → 保存为草稿或直接发布 → 同步到网关/运行时。
3. **基于表 + 手写 SQL 发布**：选表后编写参数化 SQL → 系统校验白名单与仅 SELECT → 配置 API 元信息与入参映射 → 保存/发布。
4. **编辑/下线 API**：从列表或详情进入编辑 → 修改 SQL/字段映射或元信息 → 保存（可形成新版本）；下线时校验影响并确认后下线。
5. **分类与搜索**：在 API 列表使用业务域、数据源、标签、环境、状态筛选，以及关键字搜索。
6. **在线调试**：进入 API 详情或文档页 → 选择环境、填写入参 → 发起请求 → 查看状态码与响应。
7. **查阅文档与调用**：通过文档门户或独立文档页查看 URL、入参、出参、示例与鉴权方式；导出 OpenAPI；按文档使用 AppKey/Token 调用。
8. **调用方与鉴权**：管理员/发布者为调用方创建密钥、配置 IP 白名单；必要时禁用某调用方对某 API 的访问。

### 2.5 边界与异常（≥10 条）

1. **数据源不可用**：所选数据源已禁用或连接失败。期望：发布/编辑时提示，不允许发布或需先恢复数据源。
2. **表/视图被删除或更名**：上游库表结构变更。期望：API 调用或健康检查失败时提示「数据对象不存在或已变更」，列表/详情中标记异常，需人工修正 SQL 或字段映射。
3. **手写 SQL 越权**：SQL 访问了非白名单表。期望：提交前校验失败，提示「仅允许访问已选表及白名单内对象」。
4. **SQL 非 SELECT 或含危险关键字**：出现 UPDATE/DELETE/DROP 等。期望：校验拒绝并提示仅支持 SELECT。
5. **基于表发布无条件且无分页**：用户未选任何条件且未启用分页。期望：校验不通过，提示「至少添加一个条件或开启分页并设置单页上限」。
6. **并发编辑同一 API**：多人同时编辑并保存。期望：乐观锁或版本校验，后保存者提示「数据已被他人修改，请刷新后重试」。
7. **下线已被调用的 API**：存在调用方或近期有调用量。期望：下线前展示影响提示（若有调用量），需二次确认；下线后网关返回 410 或友好提示。
8. **调用方密钥泄露或滥用**：密钥被第三方使用或高频调用。期望：支持禁用该调用方密钥；限流按 API 生效，超出返回 429 或友好提示。
9. **在线调试超时或目标库压力大**：调试触发重查询或大结果集。期望：调试请求带超时与单次返回条数上限；可配置限流保护目标库。
10. **文档与真实 API 不一致**：发布后改动了入参/出参但未更新文档。期望：文档从 API 配置自动生成，避免手写文档不同步；若支持手写补充说明，需在界面提示「以实际 API 为准」。
11. **无权限访问 API 或文档**：未登录或角色无权限。期望：管理端未登录跳转登录；文档门户可配置为匿名只读或需登录；调用 API 时鉴权失败返回 401 及明确错误码。
12. **环境不一致导致误调生产**：调试时选错环境。期望：文档与调试界面明确标注环境（测试/生产），生产环境可加强二次确认或仅对部分角色开放。

### 2.6 非功能需求

- **性能**：API 列表在 1000 条规模下，筛选+分页 P95 ≤ 3s；单次在线调试（简单查询）P95 ≤ 10s，超时明确提示。
- **安全**：调用方密钥不明文落库、传输加密；手写 SQL 仅 SELECT、白名单校验；敏感返回字段支持脱敏配置（可后续细化）。
- **审计**：2.1 中所述操作写入审计日志（操作人、时间、对象、操作类型、关键摘要）。
- **可观测**：发布/下线/调用失败有结构化日志与错误码，便于监控与排障。

## 3) 关键决策与风险

### 3.1 关键决策

| 决策点           | 选项 A（保守）     | 选项 B（折中，推荐）     | 选项 C（进取）           |
|------------------|--------------------|--------------------------|--------------------------|
| 数据对象         | 仅表               | 表 + 视图                | 表 + 视图 + 存储过程     |
| 手写 SQL 范围    | 仅当前表           | 已选表 + 同库白名单      | 同数据源全部表/视图      |
| 版本管理         | 无多版本           | 多版本 + 推荐版本        | 多版本 + 灰度路由        |
| 文档与调试       | 单 API 文档        | 单 API + 门户 + OpenAPI  | 再 + Mock/Postman 等     |
| 限流             | 不做               | 按 API 限流              | 按 API + 按调用方        |

- **当前采纳**：方案 B（折中），与澄清问题中的已选一致。

### 3.2 风险

- **安全**：白名单维护不当会扩大 SQL 可访问范围，需规范白名单评审与变更流程。
- **一致性**：网关/运行时与本模块配置不一致会导致「文档有、实际不可用」，需明确发布流程与同步机制。
- **体验**：文档门户与管理后台分离时，需统一登录与权限体验，避免调用方困惑。
