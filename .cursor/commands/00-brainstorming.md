# /brainstorming

在进入 `/feature-clarify` 之前，先用一轮轻量但系统的头脑风暴，把「粗需求清单」扩展成：本期 scope、可选增强点、未来方向和关键风险。

## 输入（用户在命令后补充）

- 粗需求清单（必填）：
  - 若干条「功能点 / 期望效果」的自然语言描述（可以直接粘贴用户原话）。
- 背景信息（可选，但推荐）：
  - 业务背景：
  - 当前现状 / 痛点：
  - 已知约束（时间 / 技术栈 / 合规 / 兼容性 等）：
  - 相关系统 / 模块（如有）：
  - 目标用户 / 角色（如有）：
- 本轮 /brainstorming 的目标（必填一两句）：
  - 例如：「重点帮我补出必须有的细节功能点和容易被忽略的坑 / 边界 / 非功能要求，不要直接进入 /feature-clarify 或写 PRD」。

> 最小输入也可以只给出：1）粗需求清单，2）一句话说明本轮想重点想清楚什么。

## 输出（建议结构）

1) 本期 scope 摘要  
   - 一句话需求：用 1 句总结这次要解决的核心问题。  
   - 本期 in-scope：本期必须要做的能力列表（bullet 列表）。  
   - 本期明确 out-of-scope 但已识别的重要点：  
     - 本期不做，但已经识别出来且对未来有价值的能力 / 场景，建议写入未来 FEAT 或 roadmap。

2) 方案选项（2–3 套）  
   - 方案 A（保守 / MVP）：  
     - 做什么：……  
     - 不做什么：……  
     - 适用场景 / 风险：……  
   - 方案 B（中等）：在 A 的基础上增加少量增强能力。  
   - 方案 C（进取）：引入更多治理 / 自动化 / 可观测性等能力。  
   - 明确给出推荐方案，并说明推荐理由。

3) 关键风险与注意事项  
   - 从安全、稳定性、数据规模、耦合关系、运维成本等角度，列出需要特别关注的点。  
   - 标记哪些风险依赖于后续设计 / 技术方案再细化。

4) 下一步建议  
   - 若本期 scope 已清晰：  
     - 建议进入 `/feature-clarify`，并使用「本期 scope 摘要」作为 `docs/features/FEAT-xxx/01-clarification.md` 的输入基线。  
   - 若 scope 仍有关键问题未拍板：  
     - 列出待拍板的问题清单，继续保持在 /brainstorming 阶段。

## 使用方式

- 在 Cursor 中触发命令时：  
  - 打开命令面板，输入 `/brainstorming`，根据「输入」小节填写粗需求清单和本轮目标。  
  - 或者在和助手的对话中，直接用类似下面的模版说明：

```text
这是某个功能的一份「粗需求清单」：
1. ...
2. ...
3. ...

请按 `/brainstorming` 命令 + `.cursor/skills/brainstorming` 的流程执行一轮头脑风暴：
- 重点帮我补出「必须有的细节功能点」和「容易被忽略的坑 / 边界 / 非功能要求」
- 暂时不要进入 /feature-clarify，也不要生成 PRD 或改代码
```

## 约束

- 本阶段只做「想清楚」「讲清楚」，不写代码、不过早定技术细节实现。  
- 不直接修改业务文档 / 规格文件，除非你明确要求将 /brainstorming 结果落盘。  
- 输出默认中文，分点编号，方便后续直接引用到 `01-clarification.md` 或 PRD。  
- 当你已经对本期 scope 充分有把握时，再进入 `/feature-clarify`。
