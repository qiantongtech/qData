1. 背景与问题

1.1 业务背景
- 当前数据来源分散在多种异构系统中（数据库、文件、HTTP API、消息队列、数仓等），各业务线/系统采用各自的方式记录和维护数据源配置：
  - 部分以文档、表格、IM 记录等“离线方式”保存；
  - 部分散落在各个应用或任务调度系统的本地配置里；
  - 连接参数（主机、端口、库名、账号、密码/Token 等）缺乏统一规范与校验。
- 在此基础上建设元数据采集、数据同步、权限审计、数据运维等能力时，存在以下痛点：
  - 新增或变更数据源需要反复沟通参数和配置方式，缺乏统一入口；
  - 环境变更（如数据库迁移、网络变更、账号密码更新）无法及时、统一同步；
  - 当采集任务失败或数据异常时，问题定位高度依赖个人经验，很难快速回溯到具体数据源配置。

1.2 当前问题与痛点
- **新接入成本高**：
  - 每次接入一个新数据源都需要从零开始确认参数、手工配置与校验，缺乏可复用的标准模板。
- **缺少统一生命周期管理**：
  - 没有统一的“启用/禁用/废弃”状态，无法在平台层面对数据源进行治理和审计。
- **可用性不可见**：
  - 无法快速确认某数据源是否仍然可连，何时出现过失败、失败原因是什么。
- **分类与责任不清晰**：
  - 数据源归属的业务域、用途、责任人模糊，影响数据治理和资产管理。
- **安全风险突出**：
  - 敏感连接信息（密码/Token/密钥）存放在不安全位置或以明文方式存在；
  - 访问和变更缺乏统一审计，权限边界不清晰。

1.3 建设本模块的核心问题
- 如何在数据中台中提供一个统一的“数据源管理”模块：
  - 统一管理多类型数据源的基础信息、连接配置和状态；
  - 支持受控角色的新增、编辑、删除、启用/禁用、连接测试、分组/分类与查询筛选；
  - 为后续元数据采集、任务调度、数据资产目录等提供可靠、标准化的数据源台账。


2. 目标与成功指标

2.1 产品目标
- 在数据中台管理后台内上线“数据源管理”模块，构建统一数据源台账，作为数据治理与采集体系的底座能力。
- 支持对多种常用数据源类型的“注册-配置-校验-启用/禁用-删除（软删）”闭环管理。
- 为后续元数据采集、任务调度等系统提供统一的数据源引用与校验入口。
- 满足基本的安全合规要求：敏感字段加密存储、变更和敏感访问可审计。

2.2 成功指标（可度量口径）
- **覆盖度指标**
  - M1：本模块上线后，核心系统使用的线上数据源（prod 环境）至少有 80% 在 3 个月内完成迁移登记到数据源管理模块中。
- **使用与效率指标**
  - M2：新建一个标准数据库数据源（MySQL/PG/Oracle 等）的平均配置时间较当前（无统一模块）降低 30% 以上（以内部访谈 + 操作步骤统计为口径）。
  - M3：数据平台管理员在排查“任务失败是否由数据源连接问题导致”时，从“定位数据源配置 → 做连接测试”的平均时间控制在 5 分钟以内。
- **性能指标**
  - M4：在 1,000 条数据源规模下，列表查询（含常用筛选）P95 响应时间 ≤ 3 秒。
  - M5：在网络正常情况下，连接测试接口 P95 响应时间 ≤ 10 秒，超时有明确提示。
- **安全与合规指标**
  - M6：所有密码/Token/密钥类字段在数据库中不以明文存储（检查库表结构和数据样本）。
  - M7：对数据源的“新增/编辑/删除、敏感字段查看/导出”等操作在审计系统中均可按“操作人 + 时间 + 对象”维度检索到记录。


3. 非目标（本期明确不做）

- 不做细粒度操作权限控制：
  - 模块仅对少量 Admin/超级管理员开放，进入模块后默认拥有该模块内全部操作权限（查看、增删改、启用/禁用、连接测试），不在本期接入统一权限中心进行更细分的操作权限控制。
- 不做自动化健康检查 / 心跳任务：
  - 不实现周期性自动连接检测、不与任务调度的前置连通性检查进行强绑定，本期仅提供手工点击的连接测试能力。
- 不做高级数据预览 / 浏览：
  - 不提供通用数据浏览器或数据采样查询功能，仅在“连接是否可用”层面进行验证。
- 不做新增数据源类型扩展：
  - 云存储 OSS/S3、ClickHouse、ES、MongoDB 等类型的接入规划为后续迭代内容，本期只支持澄清文档中列出的核心类型。
- 不强制对接外部 CMDB / 资产管理系统：
  - 本期以内建的数据源台账为主，外部 CMDB/资产系统的对接仅作为可能后续的集成方向，不纳入本期交付范围。


4. 用户与场景（用户故事）

4.1 核心用户角色
- **数据平台管理员 / 超级管理员（Admin）**
  - 有数据中台管理后台账号，人数少且受控。
  - 负责数据源的统一管理、配置变更、问题排查与治理。
- 其他角色（数据工程师、BI 分析师、业务系统负责人等）
  - 本期不直接登录该模块，通过工单或协作流程间接提出数据源新增/修改需求。

4.2 典型用户故事
- **用户故事 U1：新增一个新的业务数据库数据源**
  - 作为数据平台管理员，我希望在统一的数据源管理模块中，以标准模板快速配置一个新的 MySQL 数据库数据源，并通过“连接测试”验证配置无误，从而减少与业务方反复确认参数的时间。
- **用户故事 U2：排查采集任务失败是否由数据源导致**
  - 作为数据平台管理员，当收到任务失败告警时，我希望能快速在数据源管理模块中定位对应数据源，查看其配置与最近的连接测试情况（通过手动重测），从而判断故障是否来自数据源连通性问题。
- **用户故事 U3：治理同一业务域下的多数据源**
  - 作为数据平台管理员，我希望能按业务域、环境、类型对数据源进行分组和筛选，看到同一业务域在 prod/非 prod 环境下的全部数据源，便于统一治理和风险评估。
- **用户故事 U4：在变更数据库凭证后统一更新配置**
  - 作为数据平台管理员，当数据库账号/密码或网络信息发生变更时，我希望能在一个统一入口中查找到所有引用该数据库的配置，并在修改连接参数后立即通过连接测试验证，降低生产事故风险。
- **用户故事 U5：审计数据源配置变更与敏感访问**
  - 作为安全/合规审计人员，我希望能够在审计系统中看到谁在什么时间对哪些数据源做了新增/编辑/删除或敏感字段查看/导出操作，以满足审计与追责的需要。


5. 功能需求（编号、优先级、描述、验收标准）

> 说明：优先级定义示例：P0=本期必做，P1=本期尽量，P2=可延期。

**FR-01 数据源列表与检索**
- 优先级：P0
- 描述：
  - 在管理后台中提供“数据源管理”列表页，展示所有已注册数据源的基础信息：
    - 名称、数据源类型、业务域、环境（prod/非 prod）、责任人、状态（启用/禁用）、被引用数量等。
  - 支持按多条件组合筛选与关键字搜索：
    - 筛选条件：业务域、环境、数据源类型、状态、标签；
    - 关键字搜索：按名称、描述模糊匹配。
- 验收标准：
  - 列表加载性能在 1,000 条数据规模下 P95 响应时间 ≤ 3 秒；
  - 筛选和搜索结果只包含满足条件的数据源，无明显遗漏/多出；
  - 每条数据源行至少展示上述关键字段，且“被引用数量”与实际下游绑定数量一致（允许以异步/定时统计方式维护）。

**FR-02 新增数据源**
- 优先级：P0
- 描述：
  - Admin 可在列表页点击“新增数据源”，进入创建表单：
    - 选择数据源类型（MySQL、PostgreSQL、Oracle、SQL Server、Hive、Kafka、FTP/SFTP、HTTP API、Doris 等）；
    - 填写基础信息：名称、业务域、环境（prod/非 prod）、责任人、标签、描述等；
    - 填写连接参数：根据不同类型展示不同字段（如主机、端口、库名、用户名、密码/Token、Topic、路径、URL 等）。
  - 表单内提供“连接测试”按钮，可在保存前验证配置正确性。
  - 保存时对必填项、格式（端口、URL 等）和部分弱校验（如避免明显将生产库配置为“非生产”环境）进行校验。
- 验收标准：
  - 未填写必填字段或格式错误时，给出明确表单校验提示，无法提交；
  - 成功保存后，新数据源出现在列表中，字段值与输入一致；
  - 当配置完成并保存后，可通过“连接测试”验证连通性（见 FR-05）。

**FR-03 编辑数据源**
- 优先级：P0
- 描述：
  - Admin 可从列表或详情页进入“编辑数据源”页面，修改：
    - 基础信息（名称、业务域、环境、责任人、标签、描述）；
    - 连接参数；敏感字段使用掩码展示，未修改时不覆盖原值。
  - 支持在编辑页面再次发起连接测试。
  - 保存时记录配置变更的审计记录（操作人、时间、变更字段摘要）。
- 验收标准：
  - 修改后刷新页面仍展示新值；
  - 审计系统中可以按数据源 ID 查到本次变更记录（包含操作人、时间、变更字段摘要）；
  - 并发编辑时，如存在版本冲突，后保存的一方收到“数据已被他人修改，请刷新后重试”的提示（可通过版本号/更新时间字段实现）。

**FR-04 启用 / 禁用数据源**
- 优先级：P0
- 描述：
  - 在列表和详情页中，对每个数据源提供“启用/禁用”操作：
    - 启用：允许被新任务/作业绑定和使用；
    - 禁用：禁止新绑定，且下游系统可据此拒绝执行依赖于该数据源的新任务（具体联动由后续集成决定）。
  - 在禁用前，如果数据源已被下游任务引用，应提示“当前有 X 个任务在使用该数据源，禁用可能影响任务运行”，并要求二次确认。
  - 状态变化记录在审计系统中。
- 验收标准：
  - 列表和详情页显示的状态与实际存储值一致；
  - 禁用被引用数据源时弹出二次确认提示，提示文案包括“引用数量或风险说明”；
  - 审计中可以看到状态切换记录（含操作人、时间、前后状态）。

**FR-05 连接测试（手动）**
- 优先级：P0
- 描述：
  - 在新增/编辑表单和详情页提供“连接测试”按钮：
    - 点击时通过后端根据数据源类型进行连通性校验（数据库直连、HTTP 调用、Kafka 连接等）；
    - 设置合理超时，避免长时间占用资源。
  - 测试成功：在前端展示“连接成功”及简单耗时信息；
  - 测试失败：展示简要错误码或友好错误文案（不暴露底层堆栈和敏感信息）。
  - 所有测试行为写入日志/审计系统，包含操作人、时间、数据源标识、结果、错误摘要。
- 验收标准：
  - 对配置正确的数据源，多次点击“连接测试”返回“成功”，结果与真实连通性一致；
  - 对故意配置错误的参数（如错误密码、错误主机），返回“失败”并给出友好文案；
  - 在日志/审计系统中可以检索到对应的测试记录；
  - P95 测试响应时间 ≤ 10 秒。

**FR-06 删除数据源（软删）**
- 优先级：P0
- 描述：
  - Admin 可对不再使用的数据源执行“删除（软删）”操作：
    - 删除前校验是否被下游任务/作业引用；
    - 若被引用，默认阻止删除并提示需先清理引用或迁移；
    - 删除后数据源在列表中不再作为“可用数据源”展示，可通过特殊筛选或审计记录查看。
  - 被软删的数据源不能再被新建任务引用，但历史任务记录仍保留其引用关系。
- 验收标准：
  - 软删后通过常规列表查询不再看到该数据源，或明确标记为“已删除/不可用”；
  - 若存在引用，则默认删除失败并弹出提示文案；
  - 审计系统中有删除操作记录。

**FR-07 分组 / 分类与标签管理**
- 优先级：P1
- 描述：
  - 支持为数据源配置：
    - 业务域/主题域；
    - 环境标签（prod/非 prod）；
    - 数据源类型（数据库、MQ、文件、API 等）；
    - 自定义标签（多选）。
  - 列表页可按上述维度进行组合筛选。
- 验收标准：
  - 创建/编辑数据源时可正常设置上述字段；
  - 筛选“某业务域 + prod + 某类型 + 某标签”时结果准确；
  - 标签支持多选且可被正确保存与展示。

**FR-08 下游引用信息展示**
- 优先级：P1
- 描述：
  - 列表中展示每个数据源的“被引用数量”（下游任务/作业数量）；
  - 在数据源详情页展示只读的“引用任务列表”，包括任务名称、所属系统/项目等基础信息；
  - 删除/禁用时通过该引用信息生成风险提示。
- 验收标准：
  - 对于已与下游系统集成的场景，“被引用数量”与下游系统记录一致（允许延迟同步，但需明确刷新策略）；
  - 在详情页可以看到引用任务列表（只读，不可在此直接编辑任务配置）；
  - 删除/禁用时的提示信息中引用数量与列表信息一致。

**FR-09 审计记录接入**
- 优先级：P0
- 描述：
  - 对以下操作写入统一审计系统：
    - 新增数据源；
    - 编辑数据源（记录变更字段摘要）；
    - 删除（软删）数据源；
    - 敏感字段查看/导出操作（如有）；
  - 支持基于数据源 ID、操作人、时间区间等条件查询。
- 验收标准：
  - 随机抽取上述操作，在审计系统中均能查到对应记录；
  - 审计记录中包含操作时间、操作者、目标对象标识、操作类型、关键信息摘要。


6. 数据与权限（字段口径、权限矩阵、审计要求）

6.1 核心数据模型（字段口径草案）
- **数据源主表（示意）**
  - id：数据源唯一标识；
  - name：数据源名称，业务可识别；
  - type：数据源类型（枚举：mysql、postgresql、oracle、sqlserver、hive、kafka、ftp、sftp、http_api、doris 等）；
  - biz_domain：业务域/主题域（枚举或字典）；
  - env：环境标识（prod / non_prod）；
  - owner：责任人（可为账号或组织标识）；
  - status：状态（enabled / disabled / deleted）；
  - tags：标签集合（字符串数组或关联表）；
  - desc：描述信息；
  - conn_config：连接配置（按类型结构化存储，敏感字段需加密）；
  - created_at / created_by；
  - updated_at / updated_by。

- **连接配置字段示例**
  - 数据库类：host、port、database、username、password(enc)、parameters；
  - Kafka：bootstrap_servers、topic、security_protocol、sasl_mechanism、sasl_username(enc)、sasl_password(enc) 等；
  - FTP/SFTP：host、port、username、password(enc)、base_path、protocol 等；
  - HTTP API：base_url、auth_type、token(enc)/app_key(enc)、timeout 等。

6.2 权限矩阵（与澄清保持一致）
- 角色：
  - Admin：数据平台管理员/超级管理员；
  - Others：其他角色，本期无直接权限。
- 操作权限矩阵：
  - 查看数据源列表与详情：Admin=是，Others=否；
  - 新增数据源：Admin=是，Others=否；
  - 编辑数据源：Admin=是，Others=否；
  - 启用/禁用：Admin=是，Others=否；
  - 删除（软删）：Admin=是，Others=否；
  - 发起连接测试：Admin=是，Others=否；
  - 查看审计记录/日志摘要：Admin=是，Others=否；
  - 查看敏感字段（仅脱敏展示）：Admin=是，Others=否（无明文查看能力）。

6.3 审计与日志要求
- 审计范围：
  - 新增、编辑、删除数据源；
  - 敏感字段查看/导出；
  - 连接测试操作（可在日志中实现，但需满足按人/时间/对象可检索）。
- 审计记录字段：
  - operator_id（操作者）、operator_name；
  - operate_time；
  - target_type（如“datasource”）、target_id、target_name；
  - action_type（create/update/delete/test/secret_view/export 等）；
  - change_summary / result_summary（变更摘要或测试结果摘要）。

> 【待确认项】是否需要对“连接测试”也纳入统一审计表，而不仅仅是日志（与安全/审计团队确认）。


7. 交互与异常（概要）

7.1 交互概要
- 管理后台左侧增加“数据源管理”菜单，进入列表页。
- 列表页：
  - 顶部提供筛选区域（业务域、环境、类型、状态、标签、关键字）；
  - 表格展示基础字段 + 被引用数量 + 操作列（编辑、启用/禁用、删除、详情、连接测试）。
- 新增/编辑弹层或页面：
  - 分为“基础信息”和“连接配置”两个区块；
  - 根据数据源类型动态展示连接配置字段；
  - 底部提供“连接测试”和“保存”按钮，测试结果通过提示条/弹窗展示。
- 详情页：
  - 展示基础信息、连接配置概要（敏感字段脱敏）、状态；
  - 可查看下游引用任务列表；
  - 可查看与该数据源相关的审计记录概要。

7.2 异常与提示文案（概要）
- 表单校验失败：
  - 在对应字段下方展示错误提示，如“必填项”、“格式不正确”、“端口必须为数字”等。
- 连接测试失败：
  - 展示统一的友好文案，例如：“连接测试失败，请检查主机地址/端口/账号密码等配置。如仍有问题，请联系运维人员”，并附简要错误码；
  - 不展示底层堆栈和敏感信息。
- 删除/禁用被引用中的数据源：
  - 弹窗文案提示“当前有 X 个任务正在使用该数据源，操作可能影响任务运行，请先确认无影响或迁移后再操作”。
- 系统异常或服务不可用：
  - 统一提示“系统繁忙，请稍后重试”，同时在日志中记录详细错误。


8. 依赖与风险

8.1 外部依赖
- 统一认证与权限系统：
  - 需要确保 Admin 角色管理后台账号可识别，便于之后扩展细粒度权限。
- 日志与审计系统：
  - 需要有可写入并可查询的审计/日志管道，支持按人/时间/对象检索。
- 任务调度/采集系统：
  - 若要准确展示“被引用数量”和引用列表，需要对接调度/采集系统的任务元数据。

8.2 主要风险
- 安全风险：
  - 加密方案和密钥管理若实现不当，可能导致敏感信息泄露。
  - 错误信息处理不规范，可能通过接口或日志泄露内部结构信息。
- 操作风险：
  - Admin 误操作禁用/删除关键生产数据源，可能导致大面积任务失败，需要加强确认与审计。
- 一致性风险：
  - 若调度/采集系统未完全对接本模块，可能出现“台账状态”与“实际使用状态”不一致的情况。
- 性能风险：
  - 大量连接测试以及对大数据组件的连接可能影响下游系统性能，需要通过限流和超时控制进行保护。

> 【待确认项】与任务调度/采集系统的对接方式与时点（本期是否只展示引用信息，不做强依赖校验）。


9. 发布策略（灰度 / 回滚 / 监控）

9.1 发布策略
- 优先在测试/预生产环境完成一轮完整联调：
  - 包含常见数据源类型（MySQL、Hive、Kafka、HTTP API 等）的配置和连接测试；
  - 与审计/日志系统打通。
- 生产环境采用“小流量试点 + 逐步迁移”的方式：
  - 先选择部分业务域/系统的数据源迁移到该模块管理；
  - 在稳定运行一段时间后，再逐步将其他业务域的数据源迁移进来。

9.2 灰度与回滚
- 灰度：
  - 可通过在路由或配置中控制只有特定管理员可访问新模块；
  - 也可并行保留旧有配置方式，在稳定期内同时维护，之后再关停旧方式。
- 回滚：
  - 若发现严重问题，可临时关闭“数据源管理”菜单入口，恢复使用原有配置方式；
  - 保证本模块的改动不影响现有任务运行（任务仍使用原有配置来源），或在设计阶段明确迁移路径和回滚路径。

9.3 监控与告警
- 监控指标：
  - 数据源列表查询接口 QPS、RT；
  - 连接测试接口 QPS、RT、失败率；
  - 后端异常日志量级。
- 告警：
  - 当连接测试失败率在短时间内异常升高时触发告警；
  - 当关键接口 RT 显著上升或错误率异常时触发告警。
