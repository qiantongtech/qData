# /feature-clarify
把“一句话需求、产品功能 backlog”在原有流程上扩展为：**可评审的澄清清单 + 功能规格草案**（不写代码）。  
默认你是最终决策者；我提供可选项，你可以选 A/B/C，也可以自定义 D。

## 输入（用户在命令后补充）
- 一句话需求：
- 业务背景：
- 现状/痛点：
- 约束（时间/技术/合规/兼容性）：
- 相关系统/模块（如有）：
- 目标用户/角色（如已知）：
- 成功标准/验收口径（如已知）：
- 优先级/紧急度（P0/P1/P2 或同等定义）：

## 输出（严格按结构）
1) 澄清问题清单（按优先级，最多 15 个）  
   - 每个关键问题**优先给出 2–4 个选项**，并标明【单选】或【多选】：  
     - 示例：【单选】数据源建模方式？  
       - A) 每个环境一个独立数据源  
       - B) 逻辑数据源 + 多环境配置  
       - C) 只区分 prod / 非 prod  
     - 只能通过 IDE 弹出的**交互式选项面板**直接点选/多选，我会自动把你的选择写回澄清文档。

2) 功能规格草案  
   - 目标 / 非目标（in-scope / out-of-scope 明确）  
   - 用户角色与权限（给出权限矩阵草案，谁能看/改/删/导出等）  
   - 主流程（>=1 条，如：新增、编辑、启用/禁用、连接测试、筛选）  
   - 边界/异常（>=10 条，覆盖删除/禁用被引用、连接失败/超时、敏感字段展示等）  
   - 非功能需求（性能 / 安全 / 审计 / 可观测性）

3) 关键决策与风险（给出可选方案，方便你直接选择）  
   - 对关键决策会给出备选（如 A) 保守方案 / B) 进取方案 / C) 折中方案），你可以：  
     - 直接选：`选 B`；  
     - 或自定义：`都不选，我要 D：xxxx`。

## 约束
- 信息不足必须提出问题，但**优先用选项形式**（单选/多选），减少你重新写长描述的负担。  
- 允许你随时**自定义/改写选项**，以你最后拍板为准。  
- 不要编造关键业务规则。  
- 本阶段聚焦需求澄清与头脑风暴，不展开开发实现细节。  
- 输出默认中文、分点编号。

## Gate（必须在输出末尾给出）
- 产物清单：建议写入 `docs/features/FEAT-xxx/01-clarification.md`
- 验收清单（Checklist）：覆盖范围/权限/主流程/边界异常/验收标准
- Gate 判定：  
  - READY_FOR_NEXT 条件（全部满足才可）：  
    1) in-scope / out-of-scope 明确  
    2) 权限矩阵草案明确（谁能看/改/删/导出等）  
    3) 主流程 >= 1 条  
    4) 边界/异常 >= 10 条  
    5) 待确认项 <= 5，且**每一项都有清晰的单选/多选备选方案，你只需选项或自定义即可拍板**
    6) 成功标准/验收口径已定义（至少包含 1 条可验证标准）
  - 否则 NOT_READY，并列出阻塞项
- 下一步建议：  
  - READY_FOR_NEXT -> 进入 `/prd-generate`（需引用 `@docs/features/FEAT-xxx/01-clarification.md`）
