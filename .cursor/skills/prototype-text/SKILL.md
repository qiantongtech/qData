# /prototype-text Skill（通用）

## 概览

本 Skill 用于在**已有 PRD** 的基础上，生成“文字版原型”（不画图、不写代码），与项目命令 `.cursor/commands/03-prototype-text.md` 配套使用。

目标：**把任何功能级 PRD，细化到“前端/后端可以直接按字段和交互实现”的粒度**。

关键要求：

- 是通用的：适用于任意功能模块，而不是某个具体业务。
- 像 `/feature-clarify` 一样：**不确定就问**，通过弹窗选项与用户多轮一问一答澄清细节。

## 适用时机（When to Use）

在以下情况，必须使用本 Skill：

- 用户执行 `/prototype-text` 或 `/03-prototype-text`，且：
  - 已引用某个功能的 PRD 文件，例如：
    - `@docs/features/FEAT-xxx/02-prd.md`
    - 或者其他项目约定的 PRD 路径；
  - 命令内容来自 `.cursor/commands/03-prototype-text.md` 或等价规范。
- 目的：生成/更新 `docs/features/FEAT-xxx/03-prototype-text.md`（或同类路径）的文字版原型。

不适用场景：

- 用户只是讨论需求或写 PRD（应该用 `/feature-clarify` 或 `/prd-generate`）。
- 用户在做技术方案/实现设计（应该用 `/tech-design`）。

## 必要输入

- **PRD 文档**：例如 `docs/features/FEAT-xxx/02-prd.md`  
  - 含：背景、目标、用户与场景、功能需求列表、数据与权限、交互与异常、依赖与风险、发布策略、任务拆解等。
- **命令定义**：`.cursor/commands/03-prototype-text.md`  
  - 规定输出结构：逐页面的名称/入口/字段/交互/状态/列表规则/验收点 + Gate 规则。
- **用户的补充说明**（可选）：
  - 例如“左侧为树形结构、右侧为列表”，“已有实现可参考”等。

若缺少 PRD 引用，必须先提醒用户补充 PRD 文件引用，不要凭空造原型。

## 高层流程

1. **读取 PRD 与命令**
   - 读取 PRD 文件（如 `02-prd.md`），理解：
     - 关键用户故事与场景；
     - 功能需求列表（FR-XX）；
     - 数据结构与权限矩阵；
     - 交互与异常章节。
   - 读取 `.cursor/commands/03-prototype-text.md` 确认输出结构与 Gate 规则。

2. **识别页面集合**
   - 根据 PRD，将功能自然拆成若干页面（示例）：
     - 列表/概览页；
     - 新增/编辑表单页或弹窗；
     - 详情页；
     - 子模块专用页（如审计记录页、配置向导页等）。
   - 页面命名应源自 PRD（不要胡编），并允许后续和用户确认/重命名。

3. **交互式澄清“不确定的设计点”**
   - 对所有**不确定**的内容，优先用 AskQuestion 工具一问一答澄清，例如：
     - 布局：是否是“左树右表”、“上下分栏”、“向导步骤条”等；
     - 表单呈现：弹窗 vs 新页面；
     - 默认排序字段与顺序；
     - 分页默认条数与可选值；
     - 是否支持批量操作/批量失败提示形式；
     - 二次确认弹窗文案强度等。
   - 设计问题时，遵守：
     - 一批 3–7 个问题，避免一次问太多；
     - 明确【单选】或【多选】；
     - 每个问题给 2–4 个选项 + “其它：我在后续回答里说明”。

4. **逐页面生成文字原型**
   - 按命令约定，对每个页面输出：
     - 页面名称 & 入口；
     - 适用角色/权限；
     - 页面结构（区块划分与布局）；
     - 字段清单（字段名/类型/默认值/校验/是否展示/是否可编辑）；
     - 交互（点击/输入/提交/取消/二次确认）与按钮 enable/disable 条件；
     - 列表规则（筛选/排序/分页/批量操作）；
     - 状态（加载/空/错/无权限/成功）；
     - ≥8 条可测试验收点（可直接转化为测试用例）。

5. **Gate 判定与输出**
   - 在文末根据 `.cursor/commands/03-prototype-text.md` 中的 Gate 规则做自检：
     - 每页有结构区块；
     - 字段清单完整（类型/默认值/校验）；
     - 状态齐（加载/空/错/无权限/成功）；
     - 按钮 enable/disable 规则明确；
     - 列表规则明确（筛选/排序/分页/批量约束）；
     - 每页 ≥8 条可测试点。
   - 写出 Gate 判定（READY_FOR_NEXT / NOT_READY）及缺失项。

## 交互原则：一问一答，不怕“烦”

### 原则 1：布局/交互不确定，必须问

- **不要**因为“这个功能看起来很简单”就跳过澄清。
- 对于以下问题，除非 PRD 已经写死，否则要主动提问：
  - 列表默认排序规则（按时间？按名称？按优先级？）；
  - 列表分页配置（默认 10 / 20 / 50？最大多少？）；
  - 表单是否用 Drawer、Dialog、独立页面；
  - 是否需要批量操作，以及批量失败时的反馈形式；
  - 是否需要固定表头、固定某些列等。

### 原则 2：按钮状态必须精确

- 对每一个按钮，至少要写清：
  - **enabled** 条件；
  - **disabled** 条件；
  - 异步操作时（加载中）是否进入 loading 状态并禁用重复点击。
- 特别注意：
  - “保存”按钮：必填项/校验通过前是否可点？
  - “提交/发布”按钮：是否依赖前置步骤完成？
  - “删除/危险操作”按钮：是否强制二次确认？

### 原则 3：列表规则要细致

- 对所有列表（主列表、详情内子列表），需明确：
  - 支持哪些筛选条件，是否多条件组合；
  - 默认排序字段与方向；
  - 是否允许切换排序字段；
  - 分页默认大小、可选页大小、最大上限；
  - 是否支持多选行，以及批量操作的约束和反馈（全部成功 / 部分失败 / 全部失败）。

### 原则 4：五态齐全

- 每个页面都要覆盖这五类状态：
  - **加载态**：数据加载中；
  - **空态**：无数据/筛选无结果；
  - **错误态**：接口失败/网络错误；
  - **无权限态**：用户权限不足；
  - **成功态**：关键操作完成后的反馈（toast/提示条等）。
